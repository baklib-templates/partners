# Partners 主题数据目录说明

本目录用于沉淀 `themes/cms/partners` 主题的参考数据与素材（案例、截图、原始站点 HTML 等），方便后续补齐配置、对齐实现与验收。

---

## 本模板定位（合作伙伴门户）

「partners」类站点在现实中跨度很大（应用市场、集成目录、专家市集、纯品牌展示等）。**本主题优先服务的形态是：合作伙伴门户（Partner Portal）**——面向 **B2B**：帮助企业**管理合作伙伴关系**，并向伙伴**分发带权限控制的资源与内容**。

实质上，它仍然是一个**资源与内容管理平台**（CMS / 资源中台的一种落地形态），只是**主要受众是合作伙伴账号**（含分级、可见范围），而不是面向终端消费者的应用商店或公开创作者社区。

### 本门户应覆盖的能力（产品级 checklist）

1. **合作伙伴目录与检索**  
   伙伴/渠道/服务商名录、筛选与详情，便于内部与伙伴侧查找与背书。

2. **资源中心**  
   为合作伙伴提供的产品资料、销售与市场**物料**、**案例**、**方案**等；通常配合下载、版本与**权限**（仅登录伙伴、仅某 tier 可见等）。

3. **在线课堂 / 伙伴成长**  
   **精品课程**、**会议/培训资料**、**知识库**文章等，支撑认证与持续赋能（可与目录中的伙伴等级联动）。

4. **伙伴活动与资讯**  
   **合作伙伴大会**、**活动与事件**、**新闻**动态，形成「目录 + 资源 + 学习 + 运营活动」的闭环。

### 刻意不作为主参考的方向（边界说明）

下列形态**不是**本模板的主定位，文中**不**再作为核心案例或验收范式展开，以免与「伙伴门户 + 权限资源」混淆：

- **纯应用市场 / 插件商店**（Install App、SKU 式上架、以第三方开发者上架为主）。
- **纯集成目录**（仅罗列 API 连接器、偏自助安装的技术集成列表）。
- **泛创作者/专家市集**（按人接单、作品集展示为主，而非企业渠道伙伴赋能）。

若需调研上述方向，请单独维护笔记；本 README 的案例与「共性」章节以下文**门户与渠道计划**为准。

## PRM 与门户构建 SaaS（行业地图）

在 B2B 软件领域，专门用来**构建和管理合作伙伴门户**的系统常被统称为 **PRM（Partner Relationship Management，合作伙伴关系管理）** 软件。这类平台往往不只提供「建站」，更常见的能力还包括**线索流转**、**物料分发**、**佣金与激励结算**等。熟悉头部产品的侧重点，有助于更清晰地定位 **Baklib Partner Portal 模板**与重型 PRM 之间的差异与组合关系。

下文按侧重点分类列举市面常见方案（**调研参照**，不表示本主题需 1:1 复刻其商业模块）。

### 纯血 PRM（渠道与代理商生态）

功能高度垂直，常见于传统制造业、网络安全与大型 SaaS 的渠道体系。

| 产品 | 官网 | 地位与要点 | 对 Baklib / 本模板的启发 |
|------|------|------------|-------------------------|
| **Kademi** | [https://www.kademi.co/](https://www.kademi.co/) | **定位**：官网宣称为面向合作伙伴生态的 PRM 平台，覆盖伙伴门户、赋能与运营等常见 PRM 叙事。**适用**：需要一套更接近「完整 PRM 平台」的企业（渠道计划成熟、伙伴分层清晰、需要围绕内容/目录/线索分发形成运营闭环）。 | 与其相比，Baklib Partner Portal 更偏「主题门户 + CMS/内容中台」的轻量实现：更容易先把**目录、资源、课堂**与权限体验落地；而线索流转、深度集成与培训/认证等模块通常属于 PRM 平台或其生态（实际能力以产品与集成方案为准），实施周期也往往更长。 |
| **Impartner** | https://www.impartner.com/ | **地位**：全球市场份额最大的独立 PRM 厂商。**核心**：门户构建器（Portal Builder）强调**千人千面**——不同等级（金牌/银牌）、不同地域的伙伴登录后，营销物料、折扣价目表与培训课程可完全不同。 | **权限控制**是门户的灵魂；模板中应对核心资源（如报价单）设置清晰的**访问门槛**，以体现专业度。 |
| **Allbound** | https://www.allbound.com/ | **地位**：许多现代 SaaS 企业采用的新锐 PRM。**核心**：极度强调 **Partner Enablement（伙伴赋能）**；门户内建 **Learning Tracks（学习路径）** 与资源中心，UX 接近现代流媒体平台，而非干瘪文件夹。 | **channel-academy**、**channel-resources** 等可对标 Allbound 的视觉与信息流，突出「内容消费」体验。 |
| **Magentrix** | https://www.magentrix.com/ | **地位**：以与 **Salesforce** 无缝集成著称。**核心**：前端门户可读 CRM 数据；伙伴可在门户内 **Deal Registration（报备商机）**，与厂家销售实时同步。 | 报备/同步属 **PRM/CRM** 深水区；Baklib 模板侧重**内容与资源面**，深度流程可通过集成承接。 |

### 生态联盟与开发者网络（Affiliate / 推荐分销）

弱化传统线下渠道「重运营」模块，强化线上联盟、追踪与（部分产品中的）分成呈现。

| 产品 | 官网 | 地位与要点 | 对 Baklib / 本模板的启发 |
|------|------|------------|-------------------------|
| **PartnerStack** | https://www.partnerstack.com/ | **地位**：B2B SaaS 中较主流的联盟与伙伴平台（如 Notion、Vercel 等案例常被引用）。**核心**：干净的 **Marketplace** 模板感 + 自动化佣金；伙伴后台（Dashboard）极简，聚焦「我的推广链接」与收益。 | **UI 克制、留白多**，适合作为 Tailwind 风格门户的参考；佣金结算等**非本模板必做项**。 |
| **Crossbeam** | https://www.crossbeam.com/ | **地位**：严格说是 **Account Mapping（客户映射）** 工具，常与门户/伙伴栈组合使用。**核心**：帮助两家 SaaS 比对客户清单找重叠，促成 **Co-selling（联合销售）**。 | 与「物料 + 学习」型门户互补；本 README 仍以**伙伴门户内容架构**为主轴。 |

### 巨头 CRM 的延伸模块

用既有 CRM / 营销云底座「搭」伙伴或社区站点，强调数据同源与流程一体。

- **Salesforce Experience Cloud**（原 Community Cloud）：https://www.salesforce.com/products/experience-cloud/overview/ — 可承载客户社区、帮助中心、合作伙伴门户等；与 Salesforce CRM **原生一体**，但实施与许可成本通常较高、配置周期偏长。
- **HubSpot CMS Hub / 伙伴与生态门户**：依托入站营销与内容体系，强调用内容与物料**追踪伙伴活跃度**（打开、下载、再营销等），与「内容驱动门户」路径相近，但商业模块仍绑定 HubSpot 栈。

### 销售赋能与内容分发（最接近 Baklib「CaaS / 内容中台」叙事）

不少企业并不自称为 PRM，但用其向渠道/销售伙伴**发物料、控版本、做检索**——因传统 PRM 自带的内容模块往往较弱。

- **Highspot**：https://www.highspot.com/  
- **Seismic**：https://www.seismic.com/  

二者为 **Sales Enablement（销售赋能）** 领域常见的头部产品（业内常称「双寡头」级玩家）。**核心逻辑**：管理营销物料并追踪内容转化——销售或伙伴检索最新对客 PPT 等材料，发送给客户后，系统可追踪客户在**哪一页停留多久**等。对 Baklib 的启发：在 B2B 销售链路中，**「结构化、带标签、易检索的内容」**商业价值高；本模板应把资源中心、方案与案例的**分类与发现性**做扎实（深度行为分析属产品扩展能力）。

### Baklib 的市场切入与话术（差异化）

相对动辄**年费数万美金级**的专业 PRM 或重型赋能平台，Baklib Partner Portal 可提炼为 **「轻量级的生态赋能底座」**：

许多中腰部企业（扩张期出海 SaaS、刚铺海外渠道的制造与硬科技等）预算与流程复杂度尚未需要 Impartner、Salesforce Experience Cloud 全套闭环，当下痛点往往是：**别再靠网盘、邮件碎片向代理商发产品资料**。

通过 Baklib 的 Partner Portal 模板，可对外强调：

1. **开箱即用**：实施成本与周期显著低于传统 PRM「大项目」交付。
2. **内容驱动**：依托 Baklib 底层 CMS，在**物料管理、版本控制、知识库与在线课堂**等体验上，可优于「以商机报备与佣金为主、内容为辅」的传统 PRM 默认能力。
3. **多站点融合**：门户可与主站、招聘/品牌站等**共用内容资产与治理**，减少企业信息孤岛（具体能力以 Baklib 产品版本为准）。

## 三、合作伙伴门户常见范式（模板验收参考）

以下与第二节案例一致，服务于**合作伙伴目录 + 资源中心 + 课堂/知识库 + 活动资讯**，而非应用商店转化链。

### 1. 目录优先的检索结构（Directory / Search-First）

- 伙伴名录页宜支持**搜索与多维筛选**（如区域、行业、认证等级、合作类型）。
- 与内部 CRM/PRM 的字段对齐属于实现层话题，模板侧预留**列表 + 筛选 + 详情**结构即可。

### 2. 资源与学习的内容分区

- **资源中心**：物料、案例、方案等宜按类型/产品线/场景分类，并明确**是否需要登录或特定伙伴等级**（权限提示、占位或外链至登录）。
- **在线课堂 / 知识库**：课程、回放、讲义与文章宜有**清晰层级**（系列课、单课、附件），便于伙伴学习与复训。

### 3. 列表与卡片的可读性（Grid & Card）

- 目录与资源列表普遍采用**卡片网格**：Logo 或封面、标题、一句话摘要、标签（区域/等级/类型）、状态（新课程、热门资料等）。

### 4. 转化与动线（门户语境下的「双向」）

- **已合作或受邀伙伴**：强调**登录**、**资源获取**、**培训完成**、**活动报名**等动线。
- **潜在伙伴**：Header/Footer 保留**申请成为合作伙伴**、**联系渠道/商务**等入口；一般**不以**「安装第三方应用」为主 CTA。

### 5. 详情页主次栏

- **主栏**：方案/案例/课程正文、图文、附件列表、相关推荐。
- **侧栏**：元数据（适用区域、版本日期、联系人/支持入口）、合规或免责声明链接、**相关下载**等。

### 6. SEO / GEO 与结构化数据

- 系统已默认支持页面 **`.md` 后缀地址**（及相应元数据能力），**GEO（生成式引擎优化）与结构化呈现**由平台侧统一处理，本主题**不再**在模板内维护自定义 JSON-LD 片段。
- 若需在文档层补充语义，可在页面正文或站点级设置中按 Baklib 产品能力配置，无需在 Liquid 中手写 `application/ld+json`。
